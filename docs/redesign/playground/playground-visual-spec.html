4 — Effect 1: POIDS LOURD (physics) PG-12

The logo is a rigid body with weight. Grab it, throw it, bounce it off the viewport walls. Identical capability with mouse and finger. Device tilt is a progressive enhancement, never a requirement.

B!G
↖
pointer down = grab
S1 — Grab. Pointer down on the logo attaches a spring joint to the grab point. Cursor/finger drags it with visible lag (spring, not teleport).
B!G
↗
S2 — Throw. Release applies the pointer velocity (average of the last 90ms of samples). The off-center impulse becomes a torque about the logo's core — it tumbles on itself, never about an arbitrary pivot.
B!G
⤢
S3 — Bounce. Viewport edges are elastic walls (restitution 0.72). Impact flashes the wall edge lemon for 120ms. Each wall hit counts one interaction (PG-22).
B!G
S4 — Settle. Gravity pulls it down; it rests on the floor, slightly tilted, gently breathing (idle micro-motion). Double-tap empty space = Reset.
GROS BRAS
Secret debloque
B!G
Hidden challenge "GROS BRAS". One throw whose release speed exceeds the threshold (tuned so ~1 in 15 natural throws lands it) → celebration PG-15.

PG-13 Tilt enhancement (mobile only)

PENCHEZ VOTRE TELEPHONE
Inclinez l'écran : la gravité suit. Autorisation requise sur iPhone.
ActiverPlus tard
PG-13 Opt-in card

Rules. The card appears once, after the first successful throw (never on load). "Activer" calls DeviceOrientationEvent.requestPermission() inside the tap handler (iOS ≥13 requires a user gesture; HTTPS only). Granted → device tilt (beta/gamma) rotates the gravity vector; a small gyro glyph appears in the HUD. Denied or "Plus tard" → card never returns (localStorage pg:tilt:dismissed), effect stays fully playable with straight-down gravity. Android Chrome: no permission call needed (requestPermission is absent — its presence IS the iOS detector), listener attaches directly. Desktop: card never shows. Silent-failure guard: on iOS, listeners can attach and simply never fire — if no orientation event arrives within 1s of "granted", revert to straight-down gravity and hide the gyro glyph.

5 — Effect 2: BIG BANG (destruction) PG-14

Tap the logo: it shatters into particles from the impact point, drifts, then reassembles itself. The transgressive joy of breaking the brand — and watching it come back.

B!G
tap / click the logo
S1 — Idle. The logo floats with a slow idle rotation, faintly pulsing — it looks tappable. Coach line: "Touchez le logo".
S2 — Explode. Radial impulse from the tap point; particles inherit mesh-surface positions and colors (brutal/lemon/tangerine mix). Haptic tick where supported (navigator.vibrate(35) — silently absent on iOS).
S3 — Drift. Particles float outward with damping, slight turbulence, 1.2s. A second tap here re-explodes from the new point (chainable chaos).
B!G
S4 — Reassemble. After 1.2s of drift, particles ease back to their mesh positions over 1.6s (staggered, outermost last). The logo "heals" and resumes idle, carrying a slight natural spin about its core from the tap torque.
TIMING PARFAIT
Secret debloque
B!G
Hidden challenge "TIMING PARFAIT". Re-tap within the final 400ms of reassembly (≥90% healed) → celebration PG-15.

6 — Effect 3: LUMIERE (inspect the chrome) PG-25

The home hero's magic, handed over: rotate the logo directly and watch the studio light streaks sweep across the chrome. The camera never orbits — the object turns on its core, which is exactly the home page's grammar (scroll rotates the model, the camera stays fixed).

B!G
drag to rotate
S1 — Idle. Slow ambient rotation (~0.05 rad/s) so the streaks already travel. Coach line: "Faites tourner le logo".
B!G
⟳
S2 — Spin. Drag = trackball: pointer motion maps to angular velocity about the core; horizontal drag yaws, vertical drag pitches. It follows the finger 1:1 while held.
B!G
S3 — Release. Momentum keeps it spinning, exponentially damped (half-life ≈ 1.2s) down to the idle drift — never a dead stop. The traveling streaks do the show.
B!
S4 — Zoom. Pinch / wheel dollies the camera, clamped 1.4–4.2 (home distance: 2.6). Framing stays centered on the core.
PLEIN SOLEIL
Secret debloque
B!G
Hidden challenge "PLEIN SOLEIL". Hold the logo's lit face within ~6° of the key-light axis for 800ms → a flare glint blooms, celebration PG-15.

PG-26 The studio rig — interpreted from the home, shared by every effect

Read from scene-canvas.tsx; extracted into a shared module (move-only refactor, home tests stay green — DEC-035) consumed by the hero and all three effects:

PieceExact home values (reused verbatim)
EnvironmentProcedural scene on grey 0x565b64, PMREM-baked (blur 0.02): top softbox 14×10 ×5.0; two vertical strips (left 1.6×16 ×9.0, right 1.2×16 ×8.0) — the signature traveling streaks; floor bounce 16×3 ×3.0; cool back fill 20×20 ×0.9 tinted RGB(0.55, 0.6, 0.72).
LightsKey directional 1.6 @ (3,4,5) · rim 1.0 @ (−4,2,−3) · hemisphere 0.5.
RenderACESFilmic tone mapping, antialias, alpha, DPR ≤ 2. Camera fov 42, distance 2.6.
MaterialEvery mesh forced to metalness 1.0 · roughness 0.22 · envMapIntensity 1.15 · white base.
Rig hierarchymodel (bbox-centered at origin = the core) → holder (scale 1/maxDim; rotation.y = −π/4 cancels the GLB's baked 45°) → spin group = the sole interaction target. Every torque in every effect applies about the spin origin.

7 — Playful hooks: challenge celebration & collective counter

PG-15 Challenge celebration

On unlock: a lemon thesis-sticker toast drops from the top (spring ease, −1.2deg), displaying the badge name in the display face (ASCII: GROS BRAS, TIMING PARFAIT, PLEIN SOLEIL) over "Secret debloque"; a 1s lemon/tangerine confetti burst inside the stage (particles from the effect's own system — no new library); PG-10 flips to unlocked. The toast auto-dismisses after 3.5s. prefers-reduced-motion: toast appears without drop animation, no confetti. Unlocks are client-local only (localStorage) — no account, no server write, and the challenge names are the shareable brag ("t'as trouvé le secret ?").

PG-09 Collective counter behavior

One number, aggregated across all visitors: total interactions ("logos maltraités" — throws, wall hits, explosions). Client batches counts and flushes every 15s and on visibilitychange→hidden via navigator.sendBeacon; the chip optimistically adds local counts between server refreshes. The number never blocks play: fetch failure hides the chip, play continues. Anonymous by construction — no identifier, no cookie, no fingerprint; per-IP rate limiting only (server-side, in-memory).

PG-20 Quality tier ladder

TierEntry heuristic (static signals, no new dep)Big Bang particlesDPR cap
T3 highdesktop, or ≥8 cores & deviceMemory ≥840 0002.0
T2 middefault mobile (researched range: 20–50k @30fps on 2022-24 Android mid-tier)20 0001.5
T1 low≤4 cores or deviceMemory ≤4, or watchdog demotion6 0001.0

Watchdog: averaged FPS over a rolling ≥20-frame / 3s window; below 30 → demote DPR first, particle count second (in debounced steps — DPR changes reallocate render targets), with hysteresis; after 3 flip-flops, pin the low preset. Never promotes back within the session. Sprites: small gl_PointSize with distance attenuation, depthWrite:false, mediump fragment — fill-rate, not count, is the mobile killer. POIDS LOURD and LUMIERE are tier-insensitive (one chrome mesh) except DPR. Reduced-motion is not a tier: it renders PG-19 (no simulation at all), per REQ-008.

Copy (names, hooks, share text, challenge names) lives in content[locale].playground in src/content/site.ts, keyed by slug — site.test.ts parity extends to it (matching slugs across locales, ASCII display slots). PG-22 Counter API — /api/playground/counter

GET  → 200 { total: number }                    Cache-Control: no-store
POST { effect: EffectSlug, hits: number }       hits clamped to 1..50 per call
     → 200 { total: number }
     → 429 (rate limit: checkRateLimit(ip, { limit: 12, windowMs: 60_000 }))
     → 400 (unknown slug / bad payload — zod)

Store: src/lib/playground-counter.ts
  - authoritative in-memory total, debounced flush (5s) to
    PLAYGROUND_DATA_DIR/counter.json  (atomic: tmp + rename)
  - loads the file at boot → survives deploys and restarts
  - PLAYGROUND_DATA_DIR unset (dev): memory-only, warn once
  - deploy: named volume mounted at /data/playground (first volume in
    deploy/docker-compose.yml — none exists today; env in deploy/env.template)
Route follows the handler.ts DI pattern (injected store + rate limiter),
tested like api/contact/handler.test.ts. No cookie, no IP stored — the IP
touches only the in-memory rate limiter. Nothing personal is ever persisted.
